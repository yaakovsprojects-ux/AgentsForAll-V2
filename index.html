First, the basics

An agent isn't just a chatbot with a new name.

A chatbot answers you. An agent acts: it can read your files, run commands, browse the web, and check its own work — usually several steps in a row, without you approving each one.

→

It takes real actions

Instead of just suggesting code, a coding agent will open the file, make the edit, run the tests, and tell you what happened. The action is the output, not just text describing one.

↻

It works in a loop

Plan, act, check, adjust, repeat. If a step fails, a good agent notices and tries something else — that loop is what separates "agentic" tools from autocomplete.

⌘

It lives somewhere specific

A terminal, a browser extension, an inbox, a code editor. Where it lives determines what it can touch — which is the first thing to figure out before you install one.

The actual walkthrough

From zero to your first agent, in six steps.

This is the same process whether you're installing a coding agent, a research assistant, or an inbox agent — only the specific download changes.

01

Decide what you actually want done

"I want help with AI" is too vague for an agent to be useful. "I want something to clean up my inbox every morning" or "I want help refactoring this codebase" is something you can match to a specific tool. Pick one real task before you install anything.

02

Pick the tool that matches where the work happens

Coding work → a terminal or editor agent. Writing or research → a browser-based assistant. Email and scheduling → an inbox-connected agent. The directory below is sorted exactly this way.

see: directory below
03

Install it where it needs to live

Terminal-based agents install via a package manager. For example, a common pattern looks like this:

npm install -g <agent-cli-name> # run in your terminal

Browser or desktop agents are usually a normal download-and-open install, same as any app. If a tool asks for an API key during install, that's normal — it's how it authenticates to the model running behind it.

04

Give it the smallest possible first task

Not "reorganize my entire project." Try: "list the files in this folder and tell me which ones haven't been touched in 6 months." A small, checkable task tells you fast whether the tool actually works for you, before you trust it with anything bigger.

05

Read what it did before you let it continue

Most agents will show you a plan or a diff before acting, or a log of what it just did. Read it. This is the single habit that separates people who use agents well from people who get burned by them.

06

Give it permissions gradually

Start read-only or with confirmation prompts on. Once you've seen it behave correctly on small tasks several times, widen what it's allowed to touch — your whole codebase, your real inbox, your calendar. Trust is earned in small increments, not granted on day one.

Habits that matter more than the tool

The difference between a good and bad agent session is usually you.

Two people using the identical tool get wildly different results. Here's what separates them.

✓ Be specific about the outcome, not the method

"Fix the bug where login fails after 10 minutes" works better than "look at my auth code." Tell it the symptom and the goal; let it find the method.

✓ Give it a way to check its own work

"Run the tests after" or "verify the page loads" turns a guess into a checked result. Agents that can self-verify make far fewer mistakes you have to catch later.

✓ Keep tasks scoped to one sitting

Long, sprawling instructions accumulate small misunderstandings. Break a big goal into tasks you could each explain in one sentence.

✓ Treat its first answer as a draft

Even strong agents are better at producing a solid first pass than a perfect final one. Plan one review pass into every task, not just the risky ones.

! Don't hand over financial or irreversible actions early

Sending money, deleting data, emailing someone outside your org — keep a human-confirms-first step on anything you can't undo, no matter how long you've trusted the tool.

! Watch for confident-sounding wrong answers

Agents state incorrect things with the same tone as correct things. Spot-check anything you can't easily verify yourself, especially numbers, citations, and claims about your own code or data.

Quick reference

Terms you'll run into

Context window
How much text (code, conversation, documents) the agent can "see" at once. Bigger windows mean it can work across larger codebases or longer documents without forgetting earlier parts.
Autonomous vs. assistant mode
Assistant mode asks before each action. Autonomous mode runs a multi-step task and reports back when done. Most agents let you choose per task.
MCP (Model Context Protocol)
A common standard that lets different agents connect to the same outside tools — your files, a database, a search engine — without each one needing custom code.
Token
The unit agents are billed and limited by — roughly a chunk of a word. Long tasks, large files, and long conversations all consume more of them.
Harness
The software wrapped around an AI model that gives it tools, memory, and permissions. Two agents can use similar underlying models and still feel completely different because of the harness.
Sandbox
An isolated environment an agent runs in so its actions can't directly damage your real files or systems until you decide to apply them.
